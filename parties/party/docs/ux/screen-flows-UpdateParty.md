# Screen Flows: UpdateParty

**Entity:** party
**Связанные документы:** [use-case](../use-cases/uc-party-UpdateParty.md), [domain-model](../domain-model.md), [screen-flows-RegisterParty](screen-flows-RegisterParty.md)

## Обзор

UpdateParty — async-операция (внутренняя). Пользователь (администратор предприятия) обновляет данные организации в IdS. IdS отправляет событие в parties, которое обновляет кэшированные DynamicAttributes. Пользователь видит обновлённые данные при открытии карточки предприятия в реестре CrowdProj.

Визуальные экраны — общие (см. screen-flows-RegisterParty):
- **Экран «Реестр предприятий»** — список всех активных предприятий. После обновления данные в списке (название, ИНН) не меняются — они относятся к mandatory-полям, которые не обновляются через UpdateParty.
- **Экран «Карточка предприятия»** — детальная информация с обновлёнными DynamicAttributes (сфера деятельности, регион, контакты для администратора).

**Уникальное для UpdateParty:** на карточке предприятия отображается дата последнего обновления кэша (`updatedAt`).

**Количество экранов:** 1 (карточка предприятия — общий экран, описан в screen-flows-RegisterParty)

> **Примечание (MVP):** UpdateParty работает только с DynamicAttributes. CacheStatus, Inn, PartyName, RegistrationType, PartyId не изменяются (инвариант).

## Экран 1: Карточка предприятия (с обновлёнными данными)

Общий экран (полное описание — в screen-flows-RegisterParty §Экран 2). Ниже — только отличия для UpdateParty.

### Что видит (дополнительно к общему экрану)

| Элемент | Тип | Источник данных | Обязательность |
|---------|-----|-----------------|----------------|
| Дата последнего обновления кэша (updatedAt) | текст (дата/время) | бэкенд (parties) | Да |

Значения DynamicAttributes (BusinessSphere, Region, ContactEmail, ContactPhone, и др.) отображают состояние **после merge** — поля, переданные в событии из IdS, перезаписали соответствующие ключи. Поля, отсутствовавшие в событии, сохранили предыдущие значения (PATCH-семантика).

### Что делает

| Действие | Триггер | Что происходит |
|----------|---------|----------------|
| Вернуться в реестр | Кнопка «Назад» / жест назад | Возврат на Реестр предприятий |

Действия по обновлению данных выполняются в IdS — не в UI CrowdProj.

### Data Requirement

| Данные | Источник | Цель |

## Карта ошибок

UpdateParty — async-операция (событие из IdS). Пользователь не отправляет запрос в UI — ошибки обработки события не отображаются напрямую. Если обновление не произошло — пользователь видит предыдущие (старые) данные в карточке.

| Ошибка (из use-case) | Экран | Как показано |
|----------------------|-------|-------------|
| Idempotency-ключ уже обработан (дубликат события) | Экран 1 | Не видна — данные уже обновлены, дубликат игнорируется |
| Party с указанным PartyId не найдена | Экран 1 | Если Party не найдена при загрузке карточки — стандартный 404 / «Организация не найдена». Событие уходит в DLQ, требуется ручное разрешение |
| Payload не содержит DynamicAttributes или пуст | Экран 1 | Не видна — данные не изменились, пользователь видит старые значения. Событие считается успешно обработанным |
| Ошибка сохранения Party (конкурентный доступ, сбой БД) | Экран 1 | Retry до <!-- parameters: parties.party.retry_max_count --> 3 раз. Если retry успешны — данные обновлены. Если нет — данные не обновлены (старые значения), событие в DLQ |
| Ошибка публикации события из outbox | Экран 1 | Не видна — данные уже сохранены в parties. Outbox retry обрабатывается фоновым процессом |
| Idempotency-ключ обработан, но payload не совпадает | Экран 1 | Не видна — данные от первого события сохранены. Событие в DLQ, требуется ручное разрешение |
| Payload содержит Mandatory-поля (CacheStatus, Inn, PartyName, RegistrationType, PartyId) | Экран 1 | Не видна — запрещённые поля проигнорированы, DynamicAttributes обновлены. Пользователь видит корректные данные |

## Карта состояний

UpdateParty не изменяет CacheStatus (инвариант). Статус отображается как на общем экране карточки предприятия (screen-flows-RegisterParty).

| Статус (из domain-model) | Экран | Как показано | Примечание |
|--------------------------|-------|-------------|------------|
| active (CacheStatus) | Экран 1 | Зелёный бейдж «Активно» | CacheStatus не меняется через UpdateParty |
| inactive (CacheStatus) | Экран 1 | Серый бейдж «Неактивно» | Неактивное предприятие скрыто из публичного реестра. Если пользователь (администратор) всё же открыл карточку — видит серый бейдж и последние данные до перехода в inactive |
