# Screen: Карточка предприятия

**Entity:** party  
**Operation:** UpdateParty  
**Связанные документы:** [screen-flows](screen-flows-UpdateParty.md), [domain-model](../domain-model.md), [use-case](../use-cases/uc-party-UpdateParty.md)

---

## Назначение экрана

Пользователь просматривает детальную информацию о предприятии в карточке: название, ИНН, статус кэша, сферу деятельности, регион, публичные контакты организации. Данные обновляются асинхронно из IdS (UpdateParty). Редактирование данных предприятия выполняется через IdS — экран только для чтения.

---

## Поля экрана

Только то что пользователь реально видит. Экран read-only, редактирование через IdS.

| Поле | Тип (текст / ввод / выбор / toggle) | Источник (поле из domain-model) | Обязательность | Только чтение? |
|------|-------------------------------------|--------------------------------|----------------|----------------|
| Название предприятия (PartyName) | текст | PartyName (mandatory) | Да | Да |
| ИНН (Inn) | текст | Inn (mandatory) | Да | Да |
| Статус (CacheStatus) | бейдж | CacheStatus (mandatory) | Да | Да |
| Тип регистрации (RegistrationType) | текст | RegistrationType (mandatory) | Да | Да |
| Сфера деятельности (BusinessSphere) | текст | BusinessSphere (DynamicAttributes) | Нет | Да |
| Регион (Region) | текст | Region (DynamicAttributes) | Нет | Да |
| Публичный email (PublicEmail) | текст (ссылка) | PublicEmail (DynamicAttributes) | Нет | Да |
| Публичный телефон (PublicPhone) | текст (ссылка) | PublicPhone (DynamicAttributes) | Нет | Да |
| Дата создания (createdAt) | текст (дата) | createdAt (mandatory) | Да | Да |
| Дата последнего обновления кэша (updatedAt) | текст (дата/время) | updatedAt (computed) | Да | Да |

### Примечания по полям

- **PartyId:** внутренний идентификатор, не отображается на экране карточки.
- **ОГРН (Ogrn):** присутствует в domain-model (DynamicAttributes), но **не отображается** на публичной карточке предприятия согласно BR-005.
- **ContactEmail / ContactPhone:** присутствуют в domain-model (DynamicAttributes), но **не используются на этом экране**. Названия этих полей амбивалентны — из них нельзя однозначно определить, относятся они к публичным бизнес-контактам организации или к приватным административным контактам сотрудников (ПД). Для отображения на карточке предприятия требуются поля с явным указанием назначения: **PublicEmail** и **PublicPhone**. Эти поля отсутствуют в текущей domain-model — требуется их добавление (см. раздел «Проблемные зоны»).
- **PersonalDataConsent, OfferAccepted:** внутренние поля согласий/оферты, не отображаются на публичной карточке.

### Статусные бейджи (CacheStatus)

| Значение CacheStatus | Отображение |
|----------------------|-------------|
| active | Зелёный бейдж «Активно» — организация активна в IdS, данные кэша актуальны |
| inactive | Серый бейдж «Неактивно» — организация отключена/заблокирована/удалена в IdS |

> CacheStatus не изменяется через UpdateParty (инвариант use-case). Значение определяется другими операциями (DeactivatePartyCache / ReactivatePartyCache).

### Тип регистрации (RegistrationType)

| Значение RegistrationType | Отображение |
|--------------------------|-------------|
| real | «Зарегистрировано в IdS» (MVP — всегда `real`) |
| auto | «Автоматически создано» (MVP+) |

## Действия

Экран только для чтения. Пользователь не может редактировать данные на этом экране — редактирование выполняется через IdS.

| Действие | Триггер | Что происходит | Куда ведёт (следующий экран) |
|----------|---------|---------------|------------------------------|
| Вернуться в реестр | Кнопка «Назад» / жест назад | Возврат на реестр предприятий без дополнительных действий | Экран реестра предприятий |
| Просмотреть объявления предприятия | Кнопка «Объявления предприятия» | Переход на страницу со списком объявлений данной организации | Список объявлений предприятия |
| Связаться по email | Клик по PublicEmail (ссылка) | Открытие почтового клиента по умолчанию | Внешнее приложение |
| Позвонить по телефону | Клик по PublicPhone (ссылка) | Открытие набора номера | Внешнее приложение |

## Валидация

### Front-end (до отправки запроса)

Экран только для чтения. Пользователь не отправляет данные — front-end валидация не требуется.

| Проверка | Сообщение пользователю |
|----------|----------------------|
| — | — |

### Back-end (после отправки запроса)

Ошибки, которые приходят с сервера при загрузке данных карточки предприятия. Ошибки async-операции UpdateParty (событие из IdS) не отображаются напрямую — см. карту ошибок в screen-flows-UpdateParty.

| Ошибка | Сообщение | Где показать |
|--------|-----------|-------------|
| Предприятие не найдено (404) | «Предприятие не найдено. Возможно, оно было удалено или отключено.» | Полноэкранная страница + кнопка «Вернуться в реестр» |
| Ошибка сервера (500) | «Не удалось загрузить информацию о предприятии. Попробуйте снова.» | Alert в верхней части экрана |

> **UpdateParty (async):** Если событие обновления из IdS не обработалось (ошибка сохранения, party не найдена, DLQ) — пользователь видит предыдущие (старые) данные в карточке. Явное уведомление об ошибке обновления не показывается — операция async, пользователь не инициировал запрос из UI.

## Пустые состояния

| Состояние | Что видит пользователь | Действие |
|-----------|----------------------|----------|

## Проблемные зоны (требуют согласования с architect)

1. **ContactEmail / ContactPhone — амбивалентные названия.** В domain-model есть поля `ContactEmail` и `ContactPhone`, но из названия нельзя определить, относятся они к публичным бизнес-контактам или к приватным административным контактам (ПД). Согласно BR-005, публичные контакты организации должны отображаться, а ПД сотрудников — нет. **Требование:** переименовать или добавить поля с явной семантикой — `PublicEmail` / `PublicPhone` (публичные бизнес-контакты организации). Если текущие `ContactEmail` / `ContactPhone` являются административными контактами — они должны быть явно отделены.

2. **Нет полей `PublicEmail` / `PublicPhone` в domain-model.** Для отображения публичных контактов на карточке предприятия требуется добавить эти поля в DynamicAttributes или в mandatory-поля (если обязательны). Решение — на architect.
