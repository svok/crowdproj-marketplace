# Use Case: UpdateParty

**Entity:** party
**Связанные документы:** Domain Model, Business Viewpoint, Use Case Map

---

## Preconditions

Что должно быть истинно ДО вызова операции. Проверяет parties при получении события обновления профиля из IdS.

| # | Предусловие | Тип проверки |
|---|-------------|-------------|
| 1 | Party с указанным PartyId существует (загружается для обновления кэша) | State — наличие агрегата |
| 2 | Idempotency-ключ события не обработан ранее (защита от дубликатов) | Invariant — защита от повторной обработки |
| 3 | Payload события не пустой — содержит хотя бы одно поле для обновления DynamicAttributes | Field — проверка наличия изменяемых данных |

> **Инвариант:** CacheStatus не изменяется через этот UC. Изменение CacheStatus — ответственность UC DeactivatePartyCache / ReactivatePartyCache.
> **Инвариант:** Inn, PartyName, RegistrationType, PartyId не обновляются через этот UC. Эти поля неизменны для Party и задаются при создании (RegisterParty).

## Postconditions

Что гарантированно истинно ПОСЛЕ успешного выполнения.

| # | Постусловие | Тип результата |
|---|-------------|---------------|
| 1 | DynamicAttributes в Party обновлены: значения из payload перезаписывают существующие. Поля DynamicAttributes, отсутствующие в payload — не изменяются (частичное обновление, PATCH-семантика) | Field — обновление DynamicAttributes |
| 2 | Party доступна в реестре предприятий (SearchParties, GetPartyById) с обновлёнными DynamicAttributes | Field — видимость в публичных UI |
| 3 | Событие PartyCacheUpdated опубликовано с данными: partyId, dynamicAttributes (обновлённые), cacheStatus (текущее значение — контекст), updatedAt | Event — исходящее доменное событие |
| 4 | updatedAt = время обработки события | Field — метка времени обновления |

## Основной поток (Happy Path)

Пошаговая последовательность. Каждый шаг — одно атомарное действие системы.

1. Система получает событие обновления профиля организации от IdS (async — через брокер сообщений)
2. Система проверяет idempotency-ключ события — если ключ уже обработан, пропускает (игнорирует дубликат)
3. Система загружает Party по PartyId из payload события
4. Система проверяет, что payload содержит DynamicAttributes для обновления (не пустой объект)
5. Система выполняет merge: значения из payload перезаписывают соответствующие ключи в DynamicAttributes; ключи, отсутствующие в payload, не изменяются (PATCH-семантика)
6. Система устанавливает updatedAt = текущее время
7. Система сохраняет Party и запись в outbox-таблицу в рамках одной транзакции
8. Система публикует событие `PartyCacheUpdated` из outbox-таблицы

## Альтернативные потоки

Все отклонения от happy path.

| # | Ситуация | Шаг | Действие системы |
|---|----------|-----|-----------------|
| 1 | Idempotency-ключ уже обработан (дубликат события) | 2 | Система игнорирует событие — Party уже обновлена. Завершение без ошибки |
| 2 | Party с указанным PartyId не найдена | 3 | Система помечает событие как failed, помещает в DLQ. Логируется ошибка PartyNotFound — «Party с указанным ID не найдена». Требуется ручное разрешение (IdS — синхронизация данных) |
| 3 | Payload не содержит DynamicAttributes или DynamicAttributes пуст | 4 | Система пропускает событие (ничего не обновляется). Логируется предупреждение EmptyUpdatePayload — «Событие обновления не содержит DynamicAttributes для изменения». Событие считается успешно обработанным (не failed — это штатная ситуация, если IdS отправил холостое событие) |
| 4 | Ошибка сохранения Party (конкурентный доступ, сбой БД) | 7 | Система выполняет retry до <!-- parameters: parties.party.retry_max_count --> 3 раз с экспоненциальной задержкой (начальная <!-- parameters: parties.party.retry_backoff_ms --> 100ms). После 3 неудач — ошибка ConcurrencyConflict, событие помещается в dead-letter очередь |
| 5 | Ошибка публикации события из outbox | 8 | Outbox-запись остаётся в таблице со статусом pending. Фоновый процесс повторной обработки retry-ит публикацию |
| 6 | Idempotency-ключ существует, но payload не совпадает | 2 | Злоумышленник перехватил событие и отправляет его повторно с изменённым payload, но тем же IdempotencyKey. Система обнаруживает несовпадение payload'а с первым событием по этому ключу. Событие помечается как failed → DLQ. Логируется ошибка IdempotencyKeyMismatch — «Idempotency-ключ уже обработан, но payload отличается. Возможна replay-атака». Требуется ручное разрешение |
| 7 | Payload содержит Mandatory-поля (CacheStatus — отдельная колонка, Inn, PartyName, RegistrationType, PartyId), недопустимые для этого UC | 4 | Система игнорирует запрещённые Mandatory-поля. DynamicAttributes из payload обновляются (только ключи внутри DynamicAttributes-объекта). Запрещённые поля не применяются к Party. Логируется предупреждение UnexpectedFieldsInPayload — «Payload содержит Mandatory-поля, недопустимые для UpdateParty (CacheStatus, Inn, PartyName, RegistrationType, PartyId). Эти поля проигнорированы». Событие считается успешно обработанным — это защита от ошибочной маршрутизации внутри IdS, не атака |
