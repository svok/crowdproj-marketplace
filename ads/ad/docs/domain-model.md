## Aggregate: ad

Корень агрегата объявления (ad) в сервисе ads. Представляет собой объявление о продаже (direction=sell) или закупке (direction=buy) товаров/комплектующих. Является единой сущностью для обоих направлений — роль предприятия (продавец/покупатель) определяется динамически, по полю direction, а не атрибутом предприятия.

**Идентификатор:** AdId (UUID v4)
**Тип:** Aggregate Root
**Хранение:** state-sourced (Append-Only Ledger)

### Поля агрегата

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| id | AdId (UUID v4) | Mandatory | Уникальный идентификатор объявления |
| party_id | UUID | Mandatory | Владелец объявления (предприятие). Извлекается из JWT-токена, не реплицируется |
| created_by_user_id | UUID | Mandatory | ID пользователя (сотрудника), создавшего объявление. Используется для ролевой модели (сотрудник управляет только своими объявлениями) |
| direction | enum {sell, buy} | Mandatory | Направление: продажа или закупка. Не изменяется после публикации |
| category_id | UUID | Mandatory | Категория товара (ссылка на вершину дерева категорий ads/category) |
| title | string | Mandatory | Заголовок объявления |

### Параметры товара (attributes)

Поле `attributes` представляет собой **JSONB-массив** — список объектов-атрибутов товара. Каждый атрибут — это объект с метаданными. Категория (category_id) определяет шаблон: какие атрибуты обязательны, их типы, правила валидации.

#### Структура элемента attributes[]

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| `name` | string | Mandatory | Название атрибута (ключ, идентификатор в рамках шаблона категории). Пример: "ГОСТ", "Диаметр", "Материал" |
| `value` | string / number / boolean | Mandatory | Значение атрибута, как его указал пользователь. Тип определяется шаблоном категории |
| `unit` | string | Optional | Единица измерения (если value — числовой). Пример: "мм", "кг", "МПа", "т" |
| `normalized_value` | number | Optional | Значение атрибута, пересчитанное в универсальную единицу измерения (скрытое поле, не отображается пользователю). Используется системой матчинга для сравнения атрибутов sell↔buy: все значения приводятся к единому эталону (например, граммы для веса, миллиметры для длины). Нормализация выполняется на стороне сервера при создании/обновлении объявления |
| `description` | string | Optional | Описание атрибута — для отображения на UI (подсказка, разъяснение) |

**Принцип работы:**
- Категория задаёт шаблон: список допустимых `name`, их тип (`string`/`number`/`boolean`), обязательность, правила валидации (min, max, regex, enum-значения), а также эталонную единицу для нормализации
- Дочерняя категория наследует шаблон родительской и может дополнять/переопределять атрибуты
- При создании/редактировании объявления `attributes` валидируется:
  1. Каждый элемент массива — объект с полями `name`, `value` (минимум)
  2. `name` должен присутствовать в шаблоне категории
  3. `value` должен соответствовать типу и правилам валидации из шаблона
  4. Если атрибут числовой и шаблон определяет эталонную единицу — сервер вычисляет `normalized_value`
- Наследование: дочерняя категория наследует атрибуты родительской и может добавлять свои

**Пример для категории «Фланцы»:**
```json
[
  {
    "name": "ГОСТ",
    "value": "ГОСТ 12820-80",
    "description": "Межгосударственный стандарт"
  },
  {
    "name": "Диаметр",
    "value": 100,
    "unit": "мм",
    "normalized_value": 100.0
  },
  {
    "name": "Материал",
    "value": "Сталь 20"
  },
  {
    "name": "Давление",
    "value": 16,
    "unit": "МПа",
    "normalized_value": 160.0
  },
  {
    "name": "Вес",
    "value": 5,
    "unit": "т",
    "normalized_value": 5000000.0
  }
]
```

## State Machine

### Модель состояний

Объявление имеет **двухуровневую модель**:

1. **Основной статус** (status): `draft` / `active` / `archived`
2. **Независимый флаг блокировки** (blocked): `true` / `false` — применим только к активным объявлениям

**Видимость в поиске и матчинге:** `status=active AND blocked=false` — объявление отображается публично и участвует в матчинге.

### Состояния основного статуса

| Состояние | Описание |
|-----------|----------|
| draft | Черновик. Объявление сохранено, но не опубликовано. Видно только автору и администратору предприятия. Не участвует в поиске и матчинге |
| active | Активно. Объявление опубликовано, участвует в поиске и матчинге (если не заблокировано). После публикации direction не может быть изменён |
| archived | Архив. Объявление скрыто из поиска и матчинга. Доступно только для чтения. Редактирование содержимого недоступно — требуется восстановление в active |

### Флаг блокировки (blocked)

| Значение | Описание |
|----------|----------|
| false | Объявление не заблокировано (по умолчанию) |
| true | Объявление заблокировано модератором/системой. Применимо только к active. При blocked=true объявление не отображается в поиске и матчинге независимо от основного статуса. Владелец видит причину блокировки и может отредактировать содержимое, но не может опубликовать до разблокировки. **Визуально на UI:** в списке отображается как «Заблокировано: [причина]» (вычисляемое состояние), а не «Активно» |

### Разрешённые переходы

| Из | В | Триггер (операция) | Guard (условие) |
|----|---|---------------------|-----------------|
| (создание) | draft | CreateAd (as draft) | Пользователь авторизован, party_id из JWT |
| (создание) | active | CreateAd (publish) | Все обязательные поля заполнены (category_id, title), описание не содержит контактных данных |
| draft | active | PublishAd | Все обязательные поля заполнены; blocked=false (если blocked=true — публикация заблокирована) |
| active | draft | UnpublishAd | Пользователь подтвердил действие (для сотрудника — только своё объявление; администратор — любое) |
| active | archived | ArchiveAd | Пользователь подтвердил действие (для сотрудника — только своё объявление; администратор — любое) |
| archived | active | RestoreAd | Пользователь подтвердил действие (для сотрудника — только своё объявление; администратор — любое) |

### Управление флагом блокировки

| Действие | Эффект | Кто инициирует |
|----------|--------|----------------|
| BlockAd | active И blocked=false → blocked=true | Модератор / система. Указывается причина блокировки |
| UnblockAd | blocked=true → blocked=false | Только модератор / система. Если основной статус — active, объявление возвращается в поиск и матчинг. Если draft — снимается пометка блокировки |

### Правила флага блокировки

- Флаг блокировки применим **только к active** объявлениям. Черновики не блокируются
- При переводе active → draft флаг blocked **сохраняется** — при повторной публикации в active объявление остаётся заблокированным
- При переводе active → archived флаг blocked **сохраняется** — неактуален, т.к. архив не публичен
- При восстановлении archived → active флаг blocked **сохраняется** — если blocked=true, объявление остаётся невидимым в поиске

## Structural Invariants

Правила, истинные ВСЕГДА для агрегата ad, в любом основном статусе.

| Инвариант | Комментарий |
|-----------|-------------|
| `party_id` не изменяется после создания | Владелец объявления фиксирован. Смена предприятия-владельца требует создания нового объявления |
| `direction` не изменяется после публикации (status=active) | Направление (sell↔buy) неизменно. Для смены направления требуется новое объявление (BR-003, Global Domain Rules) |
| `title.Length` in [3..200] | Минимальная длина — 3 символа (осмысленность), максимальная — 200 (стандарт заголовка) |
| `description` не содержит контактных данных (телефон, email, ссылки) | Запрет размещения персональных контактных данных в тексте объявления (BR-003, Global Domain Rules, 152-ФЗ) |
| Установка `blocked=true` возможна только для объявлений в статусе `active` | Блокировка применима только к активным объявлениям. Черновики не блокируются. При переходе active→draft или active→archived флаг сохраняется, но блокирующий эффект применяется только к публичным статусам (active) — в draft/archived объявление и так непублично (BR-008, Global Domain Rules) |
| `blocked` сохраняется при смене основного статуса | При active↔draft и active↔archived флаг блокировки не сбрасывается (BR-008) |
| `status=archived` → содержимое только для чтения | Редактирование объявления в архиве недоступно. Для изменения требуется восстановление в active (BR-008) |
| `attributes` — массив объектов; каждый элемент содержит `name` и `value`; `name` должен присутствовать в шаблоне категории | Категория определяет допустимые имена атрибутов, их типы, обязательность и правила валидации. Валидация — при создании и обновлении объявления |
| `category_id` должен ссылаться на существующую вершину дерева категорий | Категория определяет шаблон параметров товара. Ссылочная целостность обеспечивается на уровне приложения (ads/category) |
| `direction=sell` и `direction=buy` — взаимоисключающие | Одно объявление не может быть одновременно продажей и закупкой |
| `blocked=true` → редактирование содержимого (title, description, attributes) разрешено; изменение `blocked` запрещено | Владелец может исправить нарушение в заблокированном объявлении, но не может снять блокировку самостоятельно. Только модератор/система (BR-008, Security Policy) |
| Для числовых атрибутов: если `unit` указан и категория задаёт эталонную единицу — `normalized_value` вычисляется автоматически | Нормализация выполняется сервером. Пользователь не заполняет normalized_value. Используется для сравнения атрибутов sell↔buy в матчинге |

### Примечание о cross-doc именовании

Поле владельца объявления названо `party_id`. Решение принято на основе бизнес-требований: объявление может быть как продажей (sell), так и закупкой (buy), поэтому семантически корректным является нейтральное `party_id`. Документы decomposition.md и business-viewpoint.md синхронизированы — заменено `seller_party_id` → `party_id`.
