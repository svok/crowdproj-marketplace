# MVP Scope — CrowdProj Marketplace
## 1. Цель MVP
Проверить, что CrowdProj Marketplace можно запустить за 3 месяца с помощью собственной LowCode платформы, и что минимальная доска объявлений (о продаже и покупке промкомплектующих) наберёт 100+ предприятий за 3 месяца при бесплатном использовании.
Гипотеза проверяется через критерии успеха (раздел 4): количество зарегистрированных предприятий, конверсия в размещение объявления и глубина наполнения каталога.
---
## 2. Входит в MVP
| Фича (что делаем) | Приоритет (P0/P1) | Статус |
|------------------|-------------------|--------|
| Регистрация предприятия + профиль (название, ИНН, контакты, сфера деятельности) + добавление сотрудников (несколько учётных записей на одно предприятие) | P0 | Планируется |
| Размещение объявления о продаже и объявления о закупке товара (комплектующих) — одна сущность с полем direction (продажа/закупка), характеристики определяются категорией товара (иерархическое дерево категорий, каждая со своим набором динамических атрибутов, без фиксированной схемы) | P0 | Планируется |
| Базовый матчинг: полнотекстовый поиск по строке + фильтры (категория, динамические атрибуты, определяемые категорией) — при создании/обновлении объявления система генерирует match-таблицу (подбирает релевантные **встречные** объявления: продажа ↔ закупка). При обновлении объявления матчи перегенерируются | P0 | Планируется |
| Список предприятий (реестр всех зарегистрированных компаний на платформе) | P1 | Планируется |
| Список объявлений предприятия (все активные объявления конкретной компании в одном месте) | P1 | Планируется |
| Email-напоминалка: если пользователь не заходил на платформу дольше настроенного периода (ежедневно / еженедельно / не присылать) — одно письмо о наличии новых матчингов. Не дайджест, не периодическая рассылка | P1 | Планируется |
> **Архитектурно:** Объявление о продаже и о закупке — одна сущность с полем `direction` (sell/buy). Набор характеристик товара — динамический EAV (Entity-Attribute-Value), определяется категорией товара. Категории образуют иерархическое дерево — каждая категория наследует и дополняет набор атрибутов родительской. Фиксированный набор атрибутов не применяется — каждый товар имеет собственный набор параметров.
> **Модель матчинга (MVP):** Матчинг — **постепенно эволюционирующий функционал**, не одномоментная фича.
> - **MVP:** полнотекстовый поиск (текстовое совпадение названий, описаний, категорий). При создании/обновлении объявления система генерирует match-таблицу (событийно, не view). Сравнение EAV-характеристик — не входит в MVP.
> - **Post-MVP (итеративно):** подключение поиска по атрибутам (EAV), затем AI-матчинг по техдокументации.
> - Механика «отклика»/«откликнуться» **отсутствует** — пользователь просматривает сгенерированные матчинги.

## 3. НЕ входит в MVP (Out of Scope)
| Фича (что не делаем) | Причина (почему не сейчас) | Когда добавим |
|----------------------|----------------------------|---------------|
| Расширенный матчинг (AI по ТЗ, парсинг спецификаций, NLP) | Не проверяет гипотезу MVP — доска объявлений работает на базовом поиске | В последующих версиях (после MVP+ и PMF) |
| Верификация контрагентов (фин. отчётность, арбитраж, ЕГРЮЛ) | Сложная интеграция, не критична для проверки гипотезы MVP | MVP+ фаза (после запуска MVP) |
| B2B-инструменты регулярных поставок (рамочные контракты, автозаявки, документооборот) | Требует много ресурсов, не проверяет гипотезу MVP | После подтверждения PMF |
| Мобильное приложение | Низкий приоритет — вся целевая аудитория работает с десктопа | После подтверждения PMF |
| Краулер для автоматического заполнения БД данными с пометкой «недостоверно» | MVP запускается с ручным наполнением; краулер — отдельная фаза после запуска | MVP+ фаза |
| Формализация цены/бюджета на объявлении | Цена зависит от множества факторов (объём, сроки, условия) — формализация нереалистична для MVP. Цена — один из динамических атрибутов категории, владелец может добавить по желанию | После PMF, при необходимости |
### Принятое решение (Decision Record): MVP vs ожидания Personas/CJM
**Контекст:** Утверждённые бизнес-документы (Personas v1.1.0, Value Proposition v1.1.0, CJM v2.0.1) описывают AI-матчинг по техдокументации как ключевой пользовательский опыт. MVP Scope исключает AI-матчинг — в MVP реализуется только поиск + фильтры + автоматическое сопоставление по категориям.
**Решение:** Осознанно жертвуем глубиной матчинга ради скорости выхода на рынок. AI-матчинг — долгосрочный moat и основа монетизации (Phase 1, Month 7–9 согласно monetization.md), но не критичен для проверки гипотезы MVP: «доска объявлений наберёт 100+ предприятий за 3 месяца».
**Gap будет закрыт:** AI-матчинг карточек товаров — Phase 1 (Month 7–9), AI-матчинг поисковых запросов — Phase 2 (Month 10–12+).
**Влияние на метрики:** Пользователи Personas/CJM, ожидающие AI-подбора, могут быть разочарованы на MVP. Риск частично митигируется: (1) явное позиционирование MVP как «доски объявлений», (2) Roadmap AI-матчинга публична для контрагентов.

## 4. Критерии успеха MVP
| Метрика (что измеряем) | Базовое значение (сейчас) | Целевое значение (успех за 3 мес) | Источник |
|------------------------|---------------------------|----------------------------------|----------|
| Количество зарегистрированных предприятий на платформе | 0 | ≥100 | Top-line growth |
| Количество активных предприятий (совершивших ≥1 действие за 30 дней) | 0 | ≥50 (health check порог: 30) | Health Check из premortem v1.0.2 |
| Конверсия регистрация → размещение первого объявления | 0% | >20% | Engagement |
| Среднее количество объявлений на одно активное предприятие | 0 | ≥5 | Depth of catalog |
